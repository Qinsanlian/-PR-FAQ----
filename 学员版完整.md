# 基于亚马逊 Working Backwards 的设计立项阶段培训材料

> **培训对象**：产品负责人、技术负责人、业务主管、项目发起人、跨职能核心成员  
> **培训目标**：在写任何代码前，用 PR/FAQ（新闻稿与常见问答）方法完成客户价值、方案差异化、风险与投入产出的前置验证，支撑 Go / No-Go / Defer（通过 / 不通过 / 延后）决策  
> **培训时长**：2 天集中工作坊 + 2~3 周实战辅导  
> **方法来源**：亚马逊公开资料中的 Working Backwards / PR/FAQ、Two-Pizza Team（两个披萨团队）原则、高标准写作与评审原则  
> **权威性说明**：本材料区分“公开资料可直接支持的做法”与“培训落地建议”；凡带有固定页数比例、轮次、时长、评分权重等内容，若无公开出处，均按培训建议处理  
> **学员版说明**：本版用于上课、练习与自查，保留方法要点与练习要求，弱化讲师主持话术

---

## 如何使用这份学员手册

这不是一份只用来阅读的讲义，而是一份要在课程中被持续填写、修改、讨论和复盘的工作手册。

建议按以下方式使用：
- 上课前：先通读模块 0~2，带着一个真实或模拟项目进入课堂。
- 上课中：在每个模块结束前，把本模块要求的“输出”补齐。
- 课后：把课堂产出的 PR、FAQ、假设清单和评审记录整理成自己的第一版立项材料。

### 本手册适合解决的核心问题
- 我们到底应该先定义什么，再开始写需求和排计划？
- 一个项目值不值得做，应该如何在开发前做更严格判断？
- 为什么团队常常做得很快，却仍然做错方向？
- 如何把含糊的想法变成可评审、可追问、可决策的文档？

### 你会在课程中形成的能力
- 用客户语言定义问题，而不是直接跳进功能方案
- 识别替代方案、差异化和关键假设
- 用 FAQ 主动暴露困难问题，而不是回避它们
- 在证据不足时，能明确说出应该 Go、No-Go 还是 Defer

## 模块 0：前置准备、证据标准与学习契约（0.5 小时）

### 本模块你要完成什么
- 带着一个真实问题进入课堂
- 理解什么叫“证据优先”
- 接受一个重要前提：这门课会刻意暴露你方案中的问题

### 0.1 学员入场条件
- 自带一个真实或模拟的产品 idea（产品构想）
- 至少准备一种证据（用户访谈、行为数据、工单、竞品资料等）
- 接受尖锐提问，允许方案被否定或延期

### 0.2 证据优先级（统一判断口径）
> 注：下表为培训建议的证据分级标准，非亚马逊官方要求，目的是帮助团队建立一致的判断依据。

| 级别 | 证据类型 | 用途 |
| :--- | :--- | :--- |
| A 级 | 一手访谈、行为数据、业务数据、法务 / 财务正式意见 | 决策优先依据 |
| B 级 | 行业报告、调研摘要、竞品资料、试点结果 | 支持判断 |
| C 级 | 团队经验、专家判断、类比案例 | 补充参考 |
| D 级 | 我觉得、应该会 | 不作为有效依据 |

> 补充说明：A 级证据为最优先依据，但并非所有决策都必须达到 A 级。对 P0（阻塞级）问题，建议至少达到 B 级证据后再做 Go 决策。

### 0.3 学习契约
- 对事不对人，只讨论客户、事实、风险、取舍
- 不用“感觉”替代“证据”
- 不追求首稿完美，但要每轮更清晰、更具体、更可验证
- 不用“先做再说”回避关键问题

### 0.4 为什么这个模块重要
很多培训从模板开始讲，但真正决定产出质量的，往往不是模板本身，而是你是否愿意接受以下三件事：

1. 你的第一想法很可能不够清楚。  
2. 你的第一版方案很可能经不起追问。  
3. 这不是坏事，反而正是设计阶段应该发生的事。

如果没有统一的证据口径，团队很容易陷入各说各话：
- 产品说用户会喜欢
- 技术说可以实现
- 业务说市场很大
- 财务说看不出回报

而 PR/FAQ 方法的第一步，就是把这些分散的主张变成可验证的判断。

### 本模块输出
- 统一的课堂规则
- 统一的证据优先级
- 每位学员的选题与初始风险意识

### 学员自查
- 我带来的项目，客户是谁？
- 我手上至少有一种证据吗？
- 我最怕别人问我的问题是什么？

## 模块 1：为什么需要 PR/FAQ？——从写功能转向定义客户价值（1 小时）

### 本模块你要完成什么
- 理解 PR/FAQ 不是写作文，而是做方向判断
- 理解 No-Go 不是失败，而是设计成果
- 理解为什么很多项目不是执行失败，而是方向失败

### 1.1 传统立项的三大误区
#### 误区一：从现有能力出发
团队常说：我们正好有这个能力、这个技术、这批资源，不如做点什么。  
问题是，组织从自己会做什么出发，而不是从客户真正需要什么出发。

#### 误区二：问题定义模糊
项目往往只有一个大目标，但没有说清：
- 客户是谁
- 具体场景是什么
- 客户今天怎么解决
- 为什么现有方案不够好
- 成功标准是什么

#### 误区三：把执行速度误当方向正确
有些团队执行很快，但越快越容易把错误方向做深。速度不是方向，投入也不是价值。

### 1.2 Working Backwards 的核心逻辑
- 从客户体验开始，反向定义产品
- 新闻稿（PR）迫使回答：客户是谁？为什么在意？现有方案有何不足？
- FAQ 暴露技术、财务、法务、依赖等难题

换句话说，它要求你先写“如果这个产品真的成功发布，客户会看到什么”，再回答“为了实现它，我们必须解决什么”。

### 1.3 公开资料中的关键事实
- Working Backwards 被描述为一种系统化评估想法并创造新产品的方法（见附录 C：信源 A）
- PR/FAQ 被描述为其核心工具（见附录 C：信源 A、信源 B）
- 多数 PR/FAQ 最终不会上线——这是机制收益，不是缺陷（见附录 C：信源 A）
- 团队写 10 版以上、会面 5 次以上并不罕见（见附录 C：信源 A）

### 1.4 学员要记住的结论
- PR/FAQ 的价值不在于提高立项通过率，而在于提高做对事的概率
- No-Go 不是失败，而是节省错误投入
- 立项前必须先回答“客户是谁、为什么会在意、我们为何值得做”

### 1.5 学习提醒
在这门课里，你要不断把自己从“解决方案思维”拉回到“客户价值思维”。

错误顺序通常是：
1. 我们想做一个功能
2. 我们再去找理由证明它有价值

正确顺序应该是：
1. 客户有一个重要问题
2. 当前替代方案不够好
3. 我们有一个显著更好的解决方案
4. 我们再评估是否值得投入

### 本模块输出
- 对 PR/FAQ 价值和 No-Go 价值的统一理解
- 明确立项前必须先回答什么

### 学员自查
- 我现在是不是已经急于写功能了？
- 我能不能不用产品名，只用客户问题解释为什么值得做？
- 如果这个项目最后被 No-Go，我能不能接受那也是一种高质量结果？

## 模块 2：PR/FAQ 的结构、适用边界与判断口径（1.5 小时）

### 本模块你要完成什么
- 清楚 PR 和 FAQ 分别解决什么问题
- 知道哪些场景值得写完整 PR/FAQ
- 能把 PR/FAQ 放在更大的工作方法中理解，而不是孤立看成一份文档

### 2.1 标准结构
| 部分 | 篇幅 | 视角 | 关键问题 |
| :--- | :--- | :--- | :--- |
| PR（新闻稿） | < 1 页 | 客户 / 媒体 | 客户、问题、差异化 |
| 外部 FAQ | 建议 1~2 页 | 客户 / 媒体 | 价格、安全、使用门槛 |
| 内部 FAQ | 建议 3~4 页 | 内部 | 技术、财务、法务、依赖、失败条件 |

> 说明：公开资料通常将 FAQ 总体建议控制在 5 页以内，而不是规定外部 FAQ 与内部 FAQ 的固定页数比例（见附录 C：信源 A、信源 B）。
> 补充说明：作为培训落地建议，实践中外部 FAQ 常见为 1~2 页，内部 FAQ 常见为 3~4 页，但比例可根据项目复杂度与评审重点调整。

### 2.2 适用边界
- **适合**：0→1 新产品、重要功能、跨部门项目、需高层判断的事项
- **不适合**：日常小需求、单点 Bug、零散想法
- **简化版**：POC（概念验证）/ Spike（技术探针）、技术平台建设（保留 PR 摘要 + 核心内部 FAQ）

### 2.3 与其他方法的关系
- 用户访谈 → 输入
- 市场分析 / TAM（总可寻址市场）估算 → 进入内部 FAQ
- 精益画布 → 辅助思考
- Agile / Scrum / Kanban → 在 PR/FAQ 之后进入执行

### 2.4 判断口径
- 不是所有想法都值得写 PR/FAQ
- 凡要钱、要人、要跨团队配合、要高层背书的项目，都值得写
- 文档不是行政动作，而是帮助团队提早面对现实的机制

### 2.5 你需要避免的误解
#### 误解一：PR/FAQ 就是产品文档模板
不是。它本质上是一个前置决策工具。

#### 误解二：FAQ 越长越专业
不是。篇幅受限正是它的强制力之一。你必须做取舍，必须说重点。

#### 误解三：只有面向外部客户的项目才需要 PR/FAQ
不是。内部流程、内部平台、内部机制优化，只要存在“使用者”，都可以用这个方法。

### 本模块输出
- 统一的 PR/FAQ 结构认知
- 适用边界判断口径

### 学员自查
- 我的项目属于适合、简化版还是不适合？
- 如果适合，我最应该先写 PR 还是先写 FAQ？
- 我是否已经清楚 PR 和 FAQ 的分工？

## 模块 3：动笔前的五个主管级拷问（2 小时）

### 本模块你要完成什么
- 在写 PR 之前，先把项目最关键的问题想清楚
- 用问题驱动写作，而不是用写作掩盖问题
- 形成一个可被追问的判断框架

### 3.1 客户是谁？今天用什么替代方案？
- 不能写所有人
- 必须列出至少 2 个当前替代方案（竞品、手工、Excel、外包服务、或“不处理，先忍着”）

这一问的关键不是“写出一个用户画像”，而是回答：
- 这个项目到底服务谁？
- 哪一类人最痛？
- 客户现在不是没有方案，而是在用别的东西凑合

### 3.2 问题有多痛？值得改变行为吗？
- 量化：时间、金钱、错误率、合规风险、情绪负担
- 追问：多久一次？谁最痛？凭什么换方案？

很多项目的问题不是“不存在价值”，而是“痛感不够高，撑不起改变行为”。

### 3.3 我们的方案为什么更好 / 更快 / 更便宜？
- 是否 meaningfully better（显著更好）？
- 差异点：成本、速度、成功率、便捷性、风险控制

这里最容易犯的错是把“功能更多”当成“价值更高”。客户真正关心的，通常不是你多了一个按钮，而是结果是否明显更好。

### 3.4 机会足够大？投入产出站得住？
- TAM、支付意愿、单位经济、前期投入、回收路径
- 内部项目：节省时间 / 人力、减少事故 / 返工

不是所有问题都值得作为项目去解决。一个问题即使真实存在，也要判断：
- 影响范围够不够大
- 组织能不能承受这笔投入
- 投入之后能不能带来足够回报

### 3.5 什么条件必须成立？最可能为什么失败？
- What needs to be true（哪些前提必须成立）？
- 这个项目最可能失败的三个原因是什么？
- 不要只写风险有哪些，要写具体失败路径，例如：客户不愿意改变习惯、技术瓶颈无法突破、成本太高导致定价无竞争力

### 3.6 证据化输出模板
- 每条结论标注：已验证 / 部分验证 / 待验证
- 记录证据等级、缺口、负责人、截止日期

### 3.7 本模块的正确产出形式
理想情况下，到本模块结束时，你应该已经写出：
- 一组清楚的客户定义
- 一组替代方案说明
- 一组问题严重程度判断
- 一组差异化说明
- 一组关键假设与失败路径

如果这些都还模糊，那么不建议急着进入 PR 写作。

### 本模块输出
- 每个项目的五问答案
- 关键假设与证据缺口
- 是否继续写 PR 的判断

### 学员自查
- 我能清楚说出客户今天用什么替代方案吗？
- 我的问题真的痛到值得客户改变行为吗？
- 如果别人问我最可能失败的三个原因，我能马上答出来吗？

## 模块 4：PR 新闻稿写作工作坊（3 小时）

### 本模块你要完成什么
- 把项目价值浓缩进一页以内
- 学会用客户语言而不是内部语言写 PR
- 让别人一读就知道：这是什么、解决谁的问题、为什么值得在意

### 4.1 PR 的六个核心组成
| 组成 | 作用 |
| :--- | :--- |
| 标题 | 让客户一眼看懂 |
| 副标题 | 客户 + 收益 |
| 摘要段 | 一句话说清价值 |
| 问题段 | 客户视角描述痛点 |
| 解决方案段 | 直连问题，优于现有方案 |
| 证言与上手 | 降低理解门槛 |

### 4.2 写作强制规则
- PR ≤ 1 页
- 使用客户语言，不用内部术语
- 写客户结果，不写功能清单
- 必须在 PR 的问题段或解决方案段中，明确写出至少 2 个客户当前使用的替代方案，并说明你的方案为何在这些替代方案之上是 meaningfully better
- 禁止更智能、更高效等无数据支撑的形容词

### 4.3 学员常见错误
- 只写方案，不写客户为何在意
- 只写愿景，不写当前替代方案
- 只写更好，不写好在哪里
- 用内部术语代替客户语言

### 4.4 你可以这样检查自己的 PR
#### 检查一：拿掉产品名，还能不能看懂价值
如果拿掉名字后，文字仍然能解释清楚客户问题与价值，通常说明你写到了本质。

#### 检查二：读完后有没有出现所以呢
如果读者的第一反应是“所以呢”，说明你写的是方案描述，不是价值主张。

#### 检查三：有没有明确对比替代方案
如果 PR 没有出现当前替代方案，它很可能仍停留在自说自话。

### 本模块输出
- 每组 1 份 PR 初稿
- 每组 1 份 PR 修订版

### 学员自查
- 我的标题是不是客户能理解的话？
- 我有没有写出现有替代方案？
- 我写的是客户结果，还是团队功能？

## 模块 5：FAQ 深度撰写（外部篇 + 内部篇）（3 小时）

### 本模块你要完成什么
- 把 PR 里未展开的关键问题写完整
- 让项目从“看起来不错”变成“经得起追问”
- 主动暴露难题，而不是回避难题

### 5.1 外部 FAQ（客户视角）
必答主题：价格、使用方式、支持、安全 / 隐私、适用边界  
要求：答案短、无空话、具体回答门槛与风险

外部 FAQ 的重点是：如果你是客户，你最担心什么？
- 会不会太贵
- 会不会太难用
- 会不会不安全
- 出了问题谁负责
- 我到底适不适合用

### 5.2 内部 FAQ（内部视角）
核心主题：
- 客户与替代方案
- 市场与规模
- 差异化
- 技术可行性
- 第三方依赖
- 数据与隐私
- 法务与合规
- 财务与单位经济
- 组织与运营
- 成功条件
- **失败条件（最可能失败的三个原因）**

### 5.3 假设管理：把想当然改写成待验证
- 禁止：用户愿意付费、技术上没问题、法务后面再看
- 正确格式：**待验证** + 验证方法 + 负责人 + 截止日期

### 5.4 内部 FAQ 必须避免的写法
- 答案只有会解决、后面再看、应该没问题，却没有具体结论、验证路径或依赖状态
- 有风险但无触发条件与应对动作
- 对法务、财务、运维、数据与隐私问题一笔带过

### 5.5 学员常见误区
#### 误区一：FAQ 是附属内容
不是。FAQ 很可能比 PR 更能决定项目能不能被认真看待。

#### 误区二：把问题写出来就够了
不够。FAQ 既要有问题，也要有答案；答案既可以是当前结论，也可以是待验证计划，但不能只是空话。

#### 误区三：越难回答的问题越往后放
恰恰相反，越难回答的问题越应该优先写出来。

### 本模块输出
- 外部 FAQ 初稿
- 内部 FAQ 初稿
- 假设清单与验证计划

### 学员自查
- 我的 FAQ 是在回答真实问题，还是在写套话？
- 我是否已经写出最不想被问到的问题？
- 每个关键假设都写了验证路径吗？

## 模块 6：迭代流程与事实导向评审机制（2 小时）

### 本模块你要完成什么
- 理解为什么第一版要快，后续修订要慢
- 理解评审的目标不是卖方案，而是求真
- 学会接受评审会暴露问题，而不是期待一次通过

### 6.1 核心认知：初稿要快，定稿要慢
- 第一版 PR/FAQ 应在几小时内完成（见附录 C：信源 B）
- 高质量文档需多轮修订、跨职能讨论（数周至数月）（见附录 C：信源 A、信源 B）

初稿快，不是因为它简单，而是因为第一版的职责是暴露问题，不是解决所有问题。真正高质量的 PR/FAQ，往往是被多轮追问和重写磨出来的。

### 6.2 推荐迭代流程
| 阶段 | 时长 | 性质 |
| :--- | :--- | :--- |
| 作者起草 | 数小时 | 公开资料支持 |
| 小范围打磨 | 2~5 天 | 落地建议 |
| 跨职能评审 | 1~3 轮 | 落地建议 |
| 主管 / 高层评审 | 1~2 轮 | 落地建议 |
| 会后修订 | 1~3 天 | 落地建议 |

> 注：上述阶段名称以及“2~5 天”“1~3 轮”“1~2 轮”等时长或轮次为培训落地建议；其中“作者起草：数小时”与公开资料表述一致（见附录 C：信源 B）。

### 6.3 标准评审会（60 分钟）
- 沉默阅读 15~20 分钟
- 阅读期间不讨论、不看手机、不离开座位
- 整体反馈 → 逐段讨论
- 资深者后发言
- 作者整理反馈并修订

### 6.4 主管决策会扩展版（90 分钟）
- 0~20′ 沉默阅读
- 阅读期间不讨论、不看手机、不离开座位
- 20~35′ 作者只讲三件事：客户、差异化、最大风险
- 35~70′ 事实质询与反证讨论
- 70~90′ 决策者给出 Go / No-Go / Defer + 行动项

### 6.5 决策结果
| 决策 | 含义 | 后续 |
| :--- | :--- | :--- |
| Go | 方向成立 | 进入执行拆解 |
| No-Go | 不做 | 归档原因，保留学习 |
| Defer | 条件未满足 | 明确补证清单和复审日期 |

### 6.6 No-Go 常见原因（公开资料）
- 差异化不足
- TAM 太小
- 投资过高
- 关键问题未解
- 优先级靠后

### 6.7 你在评审中应该如何看待被质疑
- 被质疑不等于被否定
- 文档被挑出漏洞，代表问题暴露得足够早
- 如果没有人提出关键问题，反而要警惕评审流于形式

### 本模块输出
- 一次标准评审演练记录
- 会议结论与行动项

### 学员自查
- 我是不是把评审理解成了“过会”而不是“求真”？
- 我最怕别人质疑哪一点？
- 如果项目被 Defer，我知道下一步该补什么证据吗？

## 模块 7：高标准写作与 Two-Pizza Team 的衔接（1.5 小时）

### 本模块你要完成什么
- 理解高标准不仅是写得更好，而是判断得更准
- 理解为什么小团队和清晰责任有助于执行 PR/FAQ 的结果
- 把设计阶段产出顺利交接到执行阶段

### 7.1 亚马逊的高标准四要点
1. 高标准是可教的
2. 高标准是领域相关的
3. 要能识别什么是好
4. 要明确完成高标准所需的工作量

这四点给你的启发是：
- 文档写不好，不一定是人不行，可能是标准不清楚
- 好文档不是一遍写完，而是经过多轮打磨
- 团队必须知道“好到什么程度才够”

### 7.2 Two-Pizza Team 的正确理解
- 理想少于 10 人
- **单线程所有权**（single-threaded ownership，即团队对某一清晰目标或服务拥有完整、不受其他并行任务干扰的专注责任）
- 高自主性 + 高交付责任

### 7.3 PR/FAQ 通过后的标准交接物
| 交接物 | 内容 |
| :--- | :--- |
| 最终版 PR/FAQ | 客户价值与范围边界 |
| 假设清单 | 已验证 / 待验证 |
| 风险清单 | P0 / P1 风险及缓解动作 |
| 指标定义 | 北极星指标、先行 / 反指标 |
| 依赖清单 | 跨团队、供应商、法务等 |
| 决策记录 | Go / Defer 的理由、放弃的备选方案 |

### 7.4 为什么这一模块对学员也重要
即使你不是主管，也要理解：
- 好的 PR/FAQ 不会自动变成好的执行
- 如果通过评审后责任边界不清晰，前面所有澄清都可能白做
- 如果没有 owner，项目很容易重新变成“大家都参与、没人真正负责”

### 本模块输出
- 每个项目的 owner 定义
- 交接物清单
- 执行阶段组织方式设计

### 学员自查
- 谁是这个项目的真正 owner？
- 谁负责上线后的持续运行？
- 通过评审后，哪些东西必须明确交接？

## 模块 8：Go / No-Go 之后，如何避免文档和执行脱节（1 小时）

### 本模块你要完成什么
- 理解为什么 PR/FAQ 不该在通过后被束之高阁
- 学会在执行中持续回写关键变化
- 理解 No-Go 文档也有组织价值

### 8.1 Go 之后的管理原则
- PR/FAQ 作为活文档持续维护
- 任何改变目标客户、价值主张、定价、重大依赖、关键风险的，必须回到文档更新复审

### 8.2 No-Go 的组织价值
- 不是失败，是节省错误投资
- 必须记录：为什么不做、哪些证据导致、哪些前提变化后可重审
- 形成反例资产

### 8.3 常见误区自检（10 条，详见附录）
- 把 PR/FAQ 当作文书
- 只有 PR，没有 FAQ
- 只讲愿景，不讲替代方案
- 把假设写成事实
- Go 之后不再更新文档

### 8.4 为什么这一步常被忽略
很多团队在立项通过后会默认：文档阶段已经结束，接下来只剩执行。结果是：
- 客户变了，文档没变
- 范围变了，文档没变
- 风险变了，文档没变

这样 PR/FAQ 很快就会失去指导作用。真正有效的做法，是把它继续当作边界说明书和变化记录器来维护。

### 本模块输出
- 文档维护规则
- No-Go 归档建议
- 执行阶段回写机制

### 学员自查
- 如果我的项目通过了，我知道什么情况需要回写文档吗？
- 如果项目没通过，我知道应该记录什么吗？
- 我能说出自己项目最容易发生的脱节点吗？

## 模块 9：实战项目演练（贯穿工作坊，约 4 小时）

### 本模块你要完成什么
- 用真实或模拟项目走一遍完整流程
- 把前面所有方法落成真正的文档产出
- 体验被提问、被挑战、被迫澄清的全过程

### 9.1 分组原则
- 每组 4~8 人（Two-Pizza 精神）
- 角色：主笔、挑战者、决策记录人、汇报人

### 9.2 分阶段产出
| 时间 | 产出 |
| :--- | :--- |
| Day 1 中段 | 五问书面答案 |
| Day 1 结束 | PR 初稿 + 关键待验证假设 |
| Day 2 上午 | 外部 FAQ + 内部 FAQ 初稿 |
| Day 2 下午 | 修订后完整 PR/FAQ + 决策建议 |

### 9.3 展示要求（每组 10 分钟）
- 客户是谁
- 问题为什么值得解决
- 方案优于替代方案
- 最大的 3 个待验证点
- 建议 Go / No-Go / Defer 的理由

### 9.4 评分维度（权重）
- 客户与问题定义清晰度 25%
- 差异化与价值主张 15%
- FAQ 完整度与难题暴露程度 25%
- 证据质量 20%
- 决策建议严谨度 15%

> 注：以上为培训建议的评分方式，供演练参考。

### 9.5 这部分最重要的不是赢，而是暴露问题
在演练中，你不需要追求让项目看起来完美。更重要的是：
- 你的问题有没有被写出来
- 你的假设有没有被标出来
- 你的决策建议有没有根据

一个最终建议 No-Go 的项目，也可能是高质量成果。

### 本模块输出
- 一套完整 PR/FAQ
- 评审与修订记录
- Go / No-Go / Defer 建议

### 学员自查
- 我们组是在讲故事，还是在做判断？
- 我们的 FAQ 里最强的一条是什么？最弱的一条是什么？
- 如果必须现在给决策，我会建议什么，为什么？

## 模块 10：落地机制、模板与后续辅导（0.5 小时）

### 本模块你要完成什么
- 理解培训结束后，如何把方法用到真实工作中
- 明确培训不是结束，而是立项机制落地的开始
- 形成自己的课后行动计划

### 10.1 组织制度落地建议
- 对需资源审批的新项目，要求先提交 PR/FAQ
- 建立分级评审：小项目（直属主管）、中项目（跨职能）、大项目（高层）
- 建立归档库：Go / No-Go / Defer 三类样本
- 每季度复盘：叫停的项目、提前识别的关键风险

### 10.2 模板与工具
| 用途 | 推荐工具 |
| :--- | :--- |
| 文档撰写 | Notion / Confluence / Google Docs |
| 证据管理 | Sheets / Airtable |
| 评审协作 | 评论 + 会议纪要模板 |
| 归档 | 团队知识库 |

### 10.3 培训效果评估
- **反应层**：满意度
- **学习层**：PR/FAQ 样本评分
- **行为层**：90 天内，符合适用边界（即要钱、要人、要跨团队配合、要高层背书）的新项目中，提交 PR/FAQ 作为立项前置材料的比例
- **结果层**：方向错误导致的重大返工是否下降

### 10.4 后续辅导计划
- 第 1 周：真实项目 PR/FAQ 初稿
- 第 2 周：线上互评
- 第 3~4 周：第一次主管评审
- 第 6~8 周：深度复盘，形成样板

### 10.5 你最应该带走的行动
课后不要只带走一份方法笔记，更要带走以下三件事：
1. 一个你准备真正去写的项目
2. 一组你知道还没验证的关键假设
3. 一个你愿意拿去接受评审的 PR/FAQ 初稿

### 本模块输出
- 组织落地动作清单
- 30~90 天推进建议
- 成效衡量方式

## 附录 A：主管评审十问（学员自查清单）

1. 客户到底是谁？不是谁？
2. 客户今天怎么解决？为什么还没被满足？
3. 问题有多痛？证据？
4. 为什么客户会改用你的方案？
5. 方案在哪些维度更好 / 更快 / 更便宜？
6. 如果对手明天也做一个，你还剩下什么？（用于检验差异化是否可持续，而非否定竞争）
7. 哪些关键能力组织今天不具备？
8. 哪三件事最可能导致失败？
9. 不做损失什么？做错损失什么？
10. 你建议 Go、No-Go 还是 Defer？依据？

建议把这十问当作每轮修订前的固定自查项，而不是等到评审会上才第一次被问到。

## 附录 B：事实 / 假设 / 风险标注法

| 标记 | 含义 |
| :--- | :--- |
| 事实 | 有 A / B 级证据支持 |
| 部分验证 | 有初步证据，样本 / 场景不足 |
| 待验证 | 暂无足够证据，必须补验证 |
| 高风险 | 方向成立但执行风险高 |

建议在文档中写清：
- 事实：证据来源是什么
- 假设：如何验证、谁负责、截止时间是什么
- 风险：触发条件和缓解动作是什么

示例：
- 事实：过去 90 天该流程平均每单多耗时 18 分钟。
- 假设：待验证。目标客户愿意为该功能支付额外月费。验证方式：10 家客户访谈。负责人：张三。截止时间：2026-06-20。
- 风险：高风险。若第三方接口 SLA 低于 99.5%，则可能影响承诺体验；缓解动作：准备备选方案。

## 结束语（学员版）

PR/FAQ 不是一份漂亮文档，而是一种管理纪律：在项目最容易被热情推动时，强迫自己回到客户、事实、约束和取舍。

对个人而言，它不只是文档能力训练，更是主人翁意识、判断力和协作能力的训练。

如果你能在进入开发前就把客户价值、替代方案、关键风险、投入产出和失败条件写清楚，那么你已经完成了设计阶段最重要的一部分工作。
