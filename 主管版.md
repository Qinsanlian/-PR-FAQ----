## 使用说明：如何读这份大纲

### 1. 内容分层
- **第一层：公开资料可直接支持**
  - PR/FAQ 是 Working Backwards 的核心工具
  - 先定义客户体验，再反推产品与能力建设
  - PR 通常少于 1 页，FAQ 建议 5 页以内
  - 多轮修订、沉默阅读、事实导向讨论是该方法的重要特征
  - Two-Pizza Team 不只是小团队，更强调单线程所有权和端到端责任
- **第二层：企业内训落地建议**
  - 课程时长、演练安排、问题分级、归档模板、角色卡、评分表
  - 这些是为了便于培训和组织执行而加入，不应误称为“亚马逊官方规定”

### 2. 主管在培训中的角色
- 不是“听懂模板”就结束，而是要学会 **把文档变成决策机制**
- 不是帮团队“把项目说圆”，而是帮助团队 **更早暴露假设、约束和错误方向**
- 不是默认所有项目都要过，而是通过 PR/FAQ **提高不做错事的能力**

### 3. 本文的培训术语说明
- 本文为了便于教学，把 **高强度、事实导向、可追问的 PR/FAQ 评审会** 简称为“角斗场”
- **说明**：这是本培训的教学术语，**不是亚马逊公开标准术语**

## 模块 0：前置准备、证据标准与学习契约（0.5 小时）

### 0.1 学员入场条件
| 内容 | 要求 |
| :--- | :--- |
| 选题准备 | 带来一个真实或模拟的产品 / 服务 / 流程改进 idea |
| 证据准备 | 至少准备一种证据：用户访谈、行为数据、工单、业务数据、竞品材料 |
| 心态准备 | 接受尖锐提问；允许自己的方案在评审中被否定或延期 |
| 工具准备 | 文档协作工具、计时器、评论权限、评审记录模板 |

### 0.2 证据优先级（主管必须先统一口径）
> **注**：下表为培训建议的证据分级标准，非亚马逊官方要求，旨在帮助团队建立一致的判断依据。

| 级别 | 可接受证据 | 说明 |
| :--- | :--- | :--- |
| A 级 | 一手用户访谈原话、产品行为数据、业务经营数据、法务或财务正式意见 | 决策优先采用 |
| B 级 | 公开行业报告、客户调研摘要、竞品公开资料、试点结果 | 可用于支持判断 |
| C 级 | 团队经验、专家判断、类比案例 | 只能作为补充，不得替代证据 |
| D 级 | “我觉得”“应该会”“行业都这么做” | 不作为有效依据 |

### 0.3 课程学习契约
- 对事不对人，只讨论客户、事实、风险、约束与取舍
- 不用“感觉正确”替代“证据充分”
- 不用“先做再说”回避关键问题
- 不追求首稿完美，但要求每轮修订比上一轮更清晰、更具体、更可验证

## 模块 1：为什么需要 PR/FAQ？——从“写功能”转向“定义客户价值”（1 小时）

### 1.1 传统立项的常见误区
- **误区 A：从现有能力出发**  
  “我们正好有这项技术 / 这支团队 / 这个窗口期，所以做点什么。”
- **误区 B：问题定义模糊**  
  只有目标，没有客户、场景、替代方案和成功标准。
- **误区 C：把执行速度误当方向正确**  
  团队很快开始写代码，但直到后期才发现客户并不在乎。

### 1.2 Working Backwards 的核心逻辑（公开资料可支持）
- 从 **客户体验** 开始，而不是从组织现有能力开始
- 用一份未来发布日期口吻的新闻稿，强迫作者回答：
  - 客户是谁？
  - 客户为什么会在意？
  - 现有方案哪里不够好？
  - 我们为什么值得做？
- 再用 FAQ 把困难问题写出来：技术、财务、法务、依赖、市场、组织能力、失败条件

### 1.3 主管可直接引用的公开资料事实
- Working Backwards 被亚马逊描述为一种 **系统化地评估想法并创造新产品的方法**（见附录 C：信源 A）
- PR/FAQ 被描述为其 **principal tool（核心工具）**（见附录 C：信源 A）
- 亚马逊公开材料明确强调：**先定义客户体验，再反向澄清“该建什么”**（见附录 C：信源 A）
- 公开资料指出：**多数 PR/FAQ 最终不会变成已发布产品，这不是缺陷，而是机制的收益**；因为它帮助组织在投入开发资源之前，决定哪些不该做（见附录 C：信源 A）
- 公开资料还指出：**团队写 10 版以上 PR/FAQ、与资深管理者会面 5 次以上反复打磨，并不罕见**（见附录 C：信源 A）

### 1.4 主管引导讨论
- 你最近一次“投入很快，但方向后来被证伪”的项目，最早在哪一周本可以停下来？
- 如果当时必须先写 PR/FAQ，最容易暴露的问题会是什么？
- 如果一个项目最终被 No-Go，它对组织到底是“浪费”，还是“省下了更大的浪费”？

## 模块 2：PR/FAQ 的结构、适用边界与主管判断口径（1.5 小时）

### 2.1 标准结构（公开资料可支持）
| 部分 | 建议篇幅 | 视角 | 关键问题 |
| :--- | :--- | :--- | :--- |
| 新闻稿（PR） | 少于 1 页 | 客户 / 媒体 | 客户是谁？解决什么问题？为什么会愿意采取行动？ |
| 外部 FAQ | FAQ 的一部分 | 客户 / 媒体 | 价格、使用方式、支持、安全、购买门槛 |
| 内部 FAQ | FAQ 的一部分 | 内部利益相关者 | 技术、财务、法务、运营、依赖、失败条件 |

> 说明：公开资料通常将 **FAQ 总体建议控制在 5 页以内**，而不是把外部 FAQ 与内部 FAQ 规定成固定页数比例。（见附录 C：信源 A、信源 B）
> 补充说明：作为培训落地建议，实践中外部 FAQ 常见为 1~2 页，内部 FAQ 常见为 3~4 页，但比例可根据项目复杂度与评审重点调整。

### 2.2 PR/FAQ 的适用边界
> 以下为基于亚马逊公开案例总结的常见适用场景，并非刚性规则。

#### 适合使用的场景
- 0→1 新产品 / 新服务
- 重要功能或体验改造
- 涉及多部门配合、需要投入较多资源的项目
- 需要主管明确判断“做 / 不做 / 暂缓”的事项
- 内部流程或机制创新，只要它有清晰“客户”（外部客户或内部使用者）

#### 不适合完整使用的场景
- 日常小需求
- 单点 Bug 修复
- 非决策性研究笔记
- 明显不会进入资源申请或立项决策的零散想法

#### 可使用简化版的场景
- POC / Spike：保留 PR 摘要 + 核心内部 FAQ
- 技术平台建设：把“客户”界定为内部开发团队或业务团队

### 2.3 与其他方法的关系
| 方法 | 作用 | 与 PR/FAQ 的关系 |
| :--- | :--- | :--- |
| 用户访谈 / 田野研究 | 发现问题与原话 | 是 PR/FAQ 的输入，不是替代物 |
| 市场分析 / TAM 估算 | 判断机会大小 | 应进入内部 FAQ，而不是单独存在 |
| 精益画布 | 梳理商业假设 | 可辅助思考，但 PR/FAQ 更强调客户语言与决策讨论 |
| Agile / Scrum / Kanban | 推动执行 | PR/FAQ 回答“做什么、为什么值得做”；敏捷回答“如何做、如何迭代” |

### 2.4 主管在本模块要统一的判断口径
- 不是所有想法都值得进入 PR/FAQ
- 但凡 **要钱、要人、要跨团队配合、要高层背书** 的项目，都值得先写 PR/FAQ
- 文档不是行政动作，而是 **迫使组织提早面对现实** 的机制

## 模块 3：动笔前的“五个主管级拷问”（2 小时）

> 以下五个问题的表述方式经过本培训改编，但核心检查点均可在公开资料中找到对应依据。（见附录 C：信源 B）

### 3.1 拷问一：客户是谁？他们今天用什么替代方案？
- 必须定义清楚目标客户，而不是写“所有人都能用”
- 必须写明客户当前如何解决这个问题：
  - 现有竞品
  - 手工流程
  - Excel / 微信 / 邮件 / 人工审批
  - “不处理，先忍着”
- 主管关注点：如果你没有写出现有替代方案，通常说明你对真实场景理解还不够

### 3.2 拷问二：问题到底有多痛？值得客户改变行为吗？
- 不只写“体验不好”，要写：
  - 时间损失
  - 金钱损失
  - 错误率
  - 合规风险
  - 情绪负担
- 主管追问示例：
  - 客户多久遇到一次？
  - 谁最痛？谁其实不痛？
  - 这个问题严重到值得客户换一个新方案吗？

### 3.3 拷问三：我们的方案为什么更好、更快、更便宜，或带来阶跃式体验变化？
- 公开资料中常见的审题方式是：
  - 是否 **meaningfully better**？
  - 是否比替代方案 **better / cheaper / faster**？
- 主管要求团队写清楚：
  - 我们解决的是同一个问题，还是换了一个问题？
  - 差异点到底体现在成本、速度、成功率、便捷性还是风险控制？
  - 如果只是“功能更多”，客户是否真的在意？

### 3.4 拷问四：机会是否足够大？投入产出是否站得住？
- 至少回答以下问题：
  - TAM 或影响范围有多大？
  - 客户是否愿意为此付费、迁移或投入时间？
  - 单位经济模型是否成立？
  - 前期投入是多少？回收路径是什么？
- 对内部项目，也要写清楚：
  - 预计节省多少时间 / 人力 / 错误成本
  - 是否减少事故、投诉、违规或返工

### 3.5 拷问五：什么条件必须成立？这个项目最可能为什么失败？
- 主管必须要求团队写出：
  - **What needs to be true**：哪些前提必须成立？
  - **Top three reasons this product will not succeed**：最可能失败的三个原因是什么？
- 这比“风险有哪些”更进一步，因为它强迫团队讨论 **失败机制**，而不只是列风险名词

### 3.6 证据化输出模板
| 命题 | 当前结论 | 证据 | 证据等级 | 缺口 | 负责人 | 截止日期 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| 客户是谁 | 例如：中型企业财务主管 | 12 次访谈 + 工单统计 | A | 缺少非现有客户样本 | 张三 | 2026-06-15 |
| 当前替代方案 | Excel + 邮件审批 | 访谈原话 + 流程图 | A | 缺少异常场景统计 | 李四 | 2026-06-18 |
| 支付意愿 / ROI | 初步成立 | 调研 + 历史项目对比 | B | 缺少真实试点 | 王五 | 2026-06-25 |

### 3.7 练习
- 选择一个真实项目，写出“五个主管级拷问”的书面答案
- 要求每条结论都标注：**已验证 / 部分验证 / 待验证**

## 模块 4：PR 新闻稿写作工作坊（3 小时）

### 4.1 PR 的核心组成（按公开模板整理）
| 组成 | 作用 | 主管检查点 |
| :--- | :--- | :--- |
| Heading（标题） | 让目标客户一眼看懂这是什么 | 是否客户能看懂，而不是内部术语？ |
| Subheading（副标题） | 写清客户是谁、收益是什么 | 是否精准定义客户，而不是泛化？ |
| Summary Paragraph（摘要段） | 用发布日期口吻概述产品及价值 | 是否能在几句话内说清价值？ |
| Problem Paragraph（问题段） | 从客户视角描述问题 | 是否是客户语言，而不是组织视角？ |
| Solution Paragraph（解决方案段） | 说明如何解决问题及差异化 | 是否直连问题，并说明为何优于当前方案？ |
| Quotes & Getting Started（引语与上手） | 让价值更具体，降低理解门槛 | 是否帮助读者想象采用后的体验？ |

### 4.2 写作强制规则
- **PR 控制在 1 页以内**，这是训练清晰度，不是限制表达欲
- 必须使用 **客户语言**，少用内部术语、平台名词、组织缩写
- 不能只写“功能”，必须写 **对客户的结果**
- 不能只有“更智能”“更高效”这类形容词，必须补充可验证的具体收益
- 不能回避现有替代方案
- 不能假设“所有人都会爱上这个产品”

### 4.3 主管常见追问
- 如果把产品名拿掉，这段文字是不是还能让人知道客户价值？
- 如果客户读完后只会问“所以呢？”，说明哪个段落失败了？
- 这份 PR 是在讲客户结果，还是在讲团队做了什么？

### 4.4 中文写作示例（用于训练，不代表官方模板原文）
- **弱标题**：下一代智能运营平台
- **较强标题**：让区域运营经理在 10 分钟内发现门店异常并完成派单

- **弱问题段**：当前运营流程效率较低，影响用户体验。
- **较强问题段**：区域运营经理每天要在 8 个系统中手工查看异常数据，常常到下午才发现门店缺货或价格配置错误，导致销售损失和客户投诉已经发生后才被处理。

- **弱解决方案段**：平台通过 AI 能力赋能运营决策。
- **较强解决方案段**：系统在每天 9:00 前自动汇总异常门店、按影响金额排序，并一键生成责任人派单，运营经理无需跨系统查数即可优先处理高损失门店。

### 4.5 实战演练
- 每组 30 分钟完成 PR 初稿
- 互换评审时，评审者只能问事实问题，不能只给“我喜欢 / 我不喜欢”式反馈
- 至少完成 2 轮修订

## 模块 5：FAQ 深度撰写（外部篇 + 内部篇）（3 小时）

### 5.1 外部 FAQ：客户与媒体会问什么？
| 主题 | 必答问题 |
| :--- | :--- |
| 价格 / 成本 | 怎么收费？是否有试用或迁移成本？ |
| 使用方式 | 如何开始使用？需要培训或安装吗？ |
| 支持与服务 | 遇到问题怎么求助？响应时间如何？ |
| 安全 / 隐私 | 数据如何处理？是否涉及权限、存储与共享？ |
| 适用边界 | 不适合谁用？在哪些场景下效果有限？ |

#### 外部 FAQ 写作要求
- 每个答案尽量短，避免术语堆砌
- 不写空话，如“请放心”“我们高度重视”
- 必须具体回答客户最在意的门槛与风险

### 5.2 内部 FAQ：主管与跨部门会问什么？
> 内部 FAQ 的价值，不在于“把项目说通”，而在于 **把所有难题提早写出来**。

| 主题 | 必答问题示例 |
| :--- | :--- |
| 客户与替代方案 | 客户今天用什么解决这个问题？为什么会改用我们？ |
| 市场与规模 | 需求规模多大？客户愿意为解决它付出什么？ |
| 差异化 | 相比现有方案，具体在哪些维度更好 / 更快 / 更便宜？ |
| 技术可行性 | 需要解决哪些关键技术问题？哪些今天还不会？ |
| 第三方依赖 | 依赖哪些合作方、平台、接口、供应商？若失效怎么办？ |
| 法务与合规 | 是否涉及合同、隐私、监管、商标、版权、跨境数据？ |
| 财务与单位经济 | 前期投入多少？单个客户 / 订单 / 项目的经济性如何？ |
| 组织与运营 | 谁来支持、监控、运维、培训、客服、交付？ |
| 成功条件 | 哪些假设必须成立？ |
| 失败条件 | 最可能失败的三个原因是什么？ |

### 5.3 假设管理：把“想当然”改写成“待验证”
- 禁止写法：
  - 用户愿意付费
  - 技术上应该没问题
  - 法务后面再看
- 推荐写法：
  - **待验证**：目标客户愿意为自动化结算支付每月 199 元以上。  
    **验证方法**：价格访谈 + 试点报价；样本量不少于 15 家目标客户。  
    **负责人**：张三。  
    **截止日期**：2026-06-20。

### 5.4 FAQ 的主管红线
- 只列问题，不给结论或验证路径
- 只有“计划会解决”，没有资源与负责人
- 写了风险，但不写触发条件与应对动作
- 对法务、财务、运维问题一笔带过
- 不写失败理由，只写成功愿景

### 5.5 练习
- 对同一个项目，补齐至少：
  - 3 条外部 FAQ
  - 8 条内部 FAQ
  - 3 条“待验证假设”

## 模块 6：迭代流程与事实导向评审机制（2 小时）

### 6.1 一个重要管理认知：初稿要快，定稿要慢
- 公开资料指出：**第一版 PR/FAQ 应该在几小时内完成，而不是几天**（见附录 C：信源 B）
- 同时，公开资料也指出：对重要项目来说，**高质量 PR/FAQ 往往需要多轮修订、跨职能讨论，甚至持续数周到数月**（见附录 C：信源 A、信源 B）
- 主管应明确传递：
  - 首稿不是为了正确，而是为了暴露问题
  - 修订不是返工，而是设计阶段最有价值的工作

### 6.2 推荐迭代流程
| 阶段 | 目标 | 时长建议 | 性质 |
| :--- | :--- | :--- | :--- |
| 阶段一：作者起草 | 快速成文，暴露假设 | 数小时 | 公开资料可支持 |
| 阶段二：小范围打磨 | 补证据、补 FAQ、补替代方案 | 2~5 天 | 企业落地建议 |
| 阶段三：跨职能评审 | 事实质询、补风险、改表述 | 1~3 轮 | 公开资料可支持 |
| 阶段四：主管 / 高层评审 | 做出 Go / No-Go / Defer 判断 | 1~2 轮 | 公开资料可支持 |
| 阶段五：会后修订 | 形成行动项与下一版 | 1~3 天 | 企业落地建议 |

### 6.3 标准 PR/FAQ 评审会（公开资料可支持）
- 作者在文档达到可分享状态后，组织 **1 小时会议**（见附录 C：信源 A）
- 会上先分发文档并 **沉默阅读 15~20 分钟**（见附录 C：信源 A、信源 B）
- 阅读结束后，先收集整体反馈，再逐段逐行讨论关键问题（见附录 C：信源 A）
- 较资深的与会者应 **尽量后发言**，避免过早影响其他人判断（见附录 C：信源 A）
- 会后由作者整理反馈与修订计划，进入下一轮迭代（见附录 C：信源 A）

### 6.4 本培训建议的“主管决策会”扩展版（企业落地建议）
> 以下为建议的扩展版，适用于需要做出 Go/No-Go 决策的关键评审；常规迭代评审仍可参照 6.3 的 60 分钟流程。

- **0~20 分钟**：沉默阅读
- **20~35 分钟**：作者只讲三件事：客户、差异化、最大风险
- **35~70 分钟**：事实质询与反证讨论
- **70~90 分钟**：决策者总结，给出 Go / No-Go / Defer + 行动项

### 6.5 决策结果的三种形式
| 决策 | 含义 | 后续动作 |
| :--- | :--- | :--- |
| Go | 方向成立，可以进入资源申请或执行拆解 | 明确范围、负责人、时间、风险项 |
| No-Go | 当前不值得做或不该做 | 归档原因，保留学习 |
| Defer | 方向可能成立，但关键条件未满足 | 明确补证据清单和复审日期 |

### 6.6 公开资料中可直接支持的 No-Go 常见原因
- 方案与现有产品相比 **不够差异化**（见附录 C：信源 B）
- **TAM 太小**，机会不值得投入（见附录 C：信源 B）
- **前期投资过高**，回报不确定或太慢（见附录 C：信源 B）
- 存在 **尚未解决的关键问题**，当前不可行（见附录 C：信源 B）
- 不是不做，而是 **优先级排在更重要项目之后**（见附录 C：信源 B）

### 6.7 问题分级（企业落地建议）
| 级别 | 定义 | 处理方式 |
| :--- | :--- | :--- |
| P0 | 不解决就不能做决策 | 补证据前不予 Go |
| P1 | 风险大，但有明确缓解路径 | 进入行动项并设截止日期 |
| P2 | 不阻塞当前决策，但需跟踪 | 进入后续 backlog |

## 模块 7：高标准写作与 Two-Pizza Team 的衔接（1.5 小时）

### 7.1 主管必须掌握的“高标准”四个要点（公开资料可支持）
亚马逊 2017 年股东信把高标准概括为四个要点（见附录 C：信源 D）：
1. **高标准是可教的**
2. **高标准是领域相关的**
3. **要能够识别什么是好**
4. **要明确教会团队完成这件事需要多大工作量（scope）**

#### 对 PR/FAQ 的直接启发
- 不要把文档质量不好归因于“作者水平不行”
- 要先定义“什么样的 PR/FAQ 才算过关”
- 要告诉团队：高质量文档通常需要多轮重写，而不是半天润色

### 7.2 Two-Pizza Team 的正确理解（公开资料可支持）
- Two-Pizza Team 不只是“团队小”（见附录 C：信源 C）
- 公开资料中的关键点包括：（见附录 C：信源 C）
  - **理想上少于 10 人**
  - **单线程所有权（single-threaded ownership）**
  - 团队对一个清晰服务或产品 **端到端负责**
  - 团队拥有更高自主性，但也承担更高可交付责任

### 7.3 主管在组织设计中的应用
- 不要让 20 个人共同“拥有”一个模糊项目
- 如果项目过大，应拆成若干单线程子问题，而不是无限扩人
- 评审 PR/FAQ 时，必须问清楚：
  - 谁是 owner？
  - 谁承担交付后的运行责任？
  - 哪些能力要嵌在团队里，哪些依赖外部团队？

### 7.4 PR/FAQ 通过后，交给执行团队的标准交接物
| 交接物 | 说明 |
| :--- | :--- |
| 最终版 PR/FAQ | 作为客户价值与范围边界的原始定义 |
| 假设清单 | 哪些已验证，哪些仍需验证 |
| 风险清单 | P0 / P1 风险及缓解动作 |
| 指标定义 | 北极星指标、先行指标、反指标 |
| 依赖清单 | 跨团队、供应商、法务、财务、数据依赖 |
| 决策记录 | 为什么 Go / Defer，放弃了哪些备选方案 |

## 模块 8：Go / No-Go 之后，如何避免文档和执行脱节（1 小时）

### 8.1 Go 之后的管理原则
- PR/FAQ **不是一次性审批材料**，而应继续作为“活文档”维护
- 任何会改变以下内容的事项，都应回到文档层面更新并复审：
  - 目标客户
  - 关键价值主张
  - 定价或单位经济
  - 重大依赖
  - 关键风险判断

### 8.2 No-Go 的组织价值
- No-Go 不是失败，而是降低错误投资的一种成功
- 主管要鼓励记录：
  - 为什么不做
  - 哪些证据导致不做
  - 哪些前提变化后可以重审
- 这会形成组织的“反例资产”，帮助未来更快做判断

### 8.3 常见误区与主管自检
| 误区 | 自检问题 |
| :--- | :--- |
| 把 PR/FAQ 当作文书 | 这份文档是否真的回答了最难的问题？ |
| 只有 PR，没有 FAQ | 你把困难问题写出来了吗？ |
| 只讲愿景，不讲替代方案 | 客户今天靠什么凑合解决？ |
| 只讲价值，不讲投入 | 前期投入、回收路径、依赖写清了吗？ |
| 把假设写成事实 | 每条关键判断都标明证据状态了吗？ |
| 评审变成“卖项目” | 大家是在求真，还是在站队？ |
| 只追求快，不追求清晰 | 首稿之后，你有安排有效修订吗？ |
| 团队太大、责任模糊 | 这件事到底谁负责到底？ |
| Go 之后不再更新文档 | 执行中的重大变化是否回写到 PR/FAQ？ |
| No-Go 不留痕 | 下次遇到类似项目，组织是否还会重复犯错？ |

### 8.4 主管的行为要求
- 在评审中追问证据，而不是替团队补故事
- 在文档不清晰时要求重写，而不是口头“先过了再说”
- 在关键前提未成立时，敢于 Defer 或 No-Go
- 把“写清楚”视为设计阶段的核心产出之一

## 模块 9：实战项目演练（贯穿工作坊，总计约 4 小时）

### 9.1 分组原则
- 每组建议 4~8 人，尽量贴近 Two-Pizza Team 的组织精神
- 每组必须明确：
  - 1 名主笔
  - 1 名挑战者
  - 1 名决策记录人
  - 1 名最终汇报人

### 9.2 分阶段产出
| 时间点 | 产出物 | 评审形式 |
| :--- | :--- | :--- |
| 第 1 天中段 | 五个主管级拷问书面答案 | 小组内互评 |
| 第 1 天结束 | PR 初稿 + 关键待验证假设 | 跨组评审 |
| 第 2 天上午 | 外部 FAQ + 内部 FAQ 初稿 | 事实导向评审 |
| 第 2 天下午 | 修订后完整 PR/FAQ + 决策建议 | 培训师 / 主管模拟决策 |

### 9.3 展示要求
- 每组 10 分钟，只讲：
  - 客户是谁
  - 问题为什么值得解决
  - 方案为何优于当前替代方案
  - 最大的 3 个待验证点
  - 建议 Go / No-Go / Defer 的理由

### 9.4 评分建议（企业落地建议）
| 维度 | 权重 |
| :--- | :--- |
| 客户与问题定义清晰度 | 25% |
| 差异化与价值主张 | 20% |
| FAQ 完整度与难题暴露程度 | 25% |
| 证据质量 | 20% |
| 决策建议的严谨度 | 10% |

## 模块 10：落地机制、模板与后续辅导（0.5 小时）

### 10.1 建议落地到组织制度中的动作
- 对需要资源审批的新项目，要求先提交 PR/FAQ
- 建立评审节奏：
  - 小项目：直属主管评审
  - 中项目：跨职能评审
  - 大项目：主管 / 高层决策会
- 建立归档库：保留 Go、No-Go、Defer 三类样本
- 每季度复盘：
  - 哪些项目因为 PR/FAQ 被及时叫停
  - 哪些项目通过 PR/FAQ 提前识别并解决了关键风险

### 10.2 模板与工具建议
| 用途 | 推荐工具 | 说明 |
| :--- | :--- | :--- |
| 文档撰写 | Notion / Confluence / Google Docs | 关键是评论与版本记录 |
| 证据管理 | Sheets / Airtable | 记录证据、假设、负责人、截止日期 |
| 评审协作 | 评论功能 + 会议纪要模板 | 确保问题可追踪 |
| 归档 | 团队知识库 | 保存 Go / No-Go 样本与教训 |

### 10.3 培训效果评估
- **反应层**：学员满意度与“方法可用性”反馈
- **学习层**：PR/FAQ 样本评分
- **行为层**：培训后 90 天内，进入立项评审的项目中 PR/FAQ 覆盖率
- **结果层**：项目启动后因方向错误导致的重大返工是否下降

## 附录 A：主管评审十问

1. 客户到底是谁？不是谁？
2. 客户今天怎么解决这个问题？为什么还没被现有方案满足？
3. 这个问题到底有多痛？证据是什么？
4. 为什么客户会改变行为、采用你的方案？
5. 你的方案在哪些维度更好 / 更快 / 更便宜？
6. 如果竞争对手明天也做一个类似功能，你还剩下什么？
7. 哪些关键能力今天组织并不具备？
8. 哪三件事最可能导致这个项目失败？
9. 如果现在不做，损失是什么；如果做错，损失又是什么？
10. 你建议 Go、No-Go 还是 Defer？依据是什么？

## 附录 B：建议使用的“事实 / 假设 / 风险”标注法

### 1. 事实
- 已有证据支持，可被复核
- 示例：过去 90 天该流程平均每单多耗时 18 分钟

### 2. 假设
- 当前成立与否未知，需要验证
- 示例：中型企业愿意为自动化审批额外支付 5% 服务费

### 3. 风险
- 即使方向成立，也可能影响交付、收益或合规
- 示例：关键第三方接口 SLA 不足，可能影响稳定性

### 4. 推荐标记方式
| 标记 | 含义 |
| :--- | :--- |
| 已验证 | 有 A / B 级证据支持 |
| 部分验证 | 有初步证据，但样本或场景不足 |
| 待验证 | 暂无足够证据，必须补验证 |
| 高风险 | 即使验证方向成立，执行风险仍高 |

**培训主管寄语**  
> PR/FAQ 不是一份漂亮文档，而是一种管理纪律：在组织最容易被激情和速度推动时，强迫自己回到客户、事实、约束和取舍。
